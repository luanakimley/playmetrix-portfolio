Usability Overview — General

In this page, I am going to do a usability overview on general tasks that are not specific to any roles, detailing:

  • Initial/current problems
  • Implemented solutions
  • Further enhancements/recommendations

The problems detailed here mostly arise from our user testing, but I will also do my own analysis by going through the app and some problems that I noticed while developing the frontend.

I will be recommending enhancements and solutions from both design and technical perspectives as the UI/UX Designer and Frontend Developer.

Onboarding the App

Problem

  • Some users, specifically those in the older age group had difficulties in getting onboarding with the app, as we did not have any help and/or documentation. This also violates Jakob Nielsen and Rolf Molich’s Ten User Interface Guidelines #10: Help and documentation.

Implemented Solutions

  • Introduction Screens - these set of screens will be shown upon first launch of application, giving the users a brief introduction of PlayMetrix. It also provides a link to our Linktree which has links to our social medias which they can contact us from, also with a link to our guidebook.

  • Online Tutorial - we also came up with the idea of providing an online tutorial, which is then made by Shakira. This will provide a comprehensive guidance for the manager and the physio, which are the users we found needed it the most from our user testing.
  • In-app Features Showcase - to further enhance the user's onboarding experience, I also developed a set of screens with the features of each user. Each user role in our application has different functionalities, so I ensured that these screens are personalised to showcase key features of that particular user role.

Signing Up

Problem

  • While conducting user testing, we found out that some users, also in the older age group, being overwhelmed by the sign up screen as they would need to enter a lot of information on one screen.
  • Initial sign up flow:

Implemented Solutions

  • Multi-step sign up form - Implementing a sign up form that consists of multiple screens with the user only needing to input one or two text fields would reduce the feeling of being overwhelmed. I also gave a clear indicator of the number of steps needed to finish the form so they will have a clear indicator of the progress they made.

Further Enhancements

  • A single sign on (SSO) with several different platforms would definitely be more beneficial for the user, as this will reduce the time they need for signing up and they would not need to remember additional passwords and use their existing accounts. However, due to the way our database is set up and time constraints, we could not implement this in our application for the time being.
Signing In

Problem

  • It is highlighted in the user testing that users, particularly those that are not tech-savvy are also having difficulties in signing in as they were confused on where to put in the login credentials.
  • While testing the app, we also came across the issue of the user sessions not saving, which is quite annoying and disrupt the user's workflow as they will need to sign in again every time after they close the app.

Implemented Solutions

  • Google Single Sign On (SSO) - In order to streamline the user sign in, Nathan implemented Google Sign On functionality for both the backend and frontend. After conducting more user tests, this proved to be useful as it really reduced the time the user needs to sign in.
  • Sessions - I used Flutter's shared_preferences package to implement a local session storage to save user credentials securely after sign in, so they wouldn't need to re-sign in every time after they close the app.

Further Enhancements

  • Implementing SSOs with different platforms would be something that we could look at. A research on which SSO is used the most by our user demographics would be beneficial too before implementing this.
  • Implementing auto-fill password would also be something to be considered by saving the password to Google or Apple keychain so the user can sign in easily to their account without having to recall their password.
Home Screen

Problem

  • It is pointed out that the initial design of our home screen does not really include the features that the user would use the most or what they would want to see first when they open the app.

Implemented Solutions

  • Home screen redesign - Driven from that feedback, I redesigned the home screen to be more compact and to include a crucial feature of our app, the notifications. At first glance, our users would want to know what they missed or what is happening in the team through the notifications, so it is important to highlight this in the home screen.

Further Enhancements

  • Personalised & customisable home screen - Currently, we have some menu buttons that will redirect the user to other screens, but these menu buttons are not personalised or customisable to the user. Going forward, it would be a good recommendation to further fulfil Jakob Nielsen and Rolf Molich’s Ten User Interface Guidelines #7: Flexibility and Efficiency of Use by showing the screens mostly used by the user or recently accessed ones which can be known from user testing or data analytics of the user's app usage. Having it to be customisable, with the users being able to choose the menu buttons they want shown in their home screen would also be beneficial for the flexibility aspect.
