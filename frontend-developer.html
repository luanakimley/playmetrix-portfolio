Frontend Developer

Overview

My main role in this group is as a Frontend Developer, my main job was to build the frontend side of the application, which is the part of the application the user interacts with, aiming to make it user friendly while also being aesthetically pleasing to look at.

The steps I took were to:

  1. Develop the non-functional structure of each pages.
  2. Specify what is shown on each pages based on the user role.
  3. Develop the functionalities of each pages (connecting to backend).

Technical Challenges

  • Getting used to mobile development and the frontend framework I worked with.
  • Making sure that the application is responsive and will look consistent between different devices.
  • Finding and fixing possible bugs and catch errors in the frontend to ensure smooth user experience.
  • Optimising the frontend for speed and efficiency by minimising requests and optimising files.
Frontend Technology

I chose to use Flutter with Dart programming language as the frontend technology of choice.

While developing the app, I utilised several Flutter packages, such as:

  • Syncfusion Flutter Calendar — calendar views
  • Riverpod — state management
  • Image Picker — picking and uploading images
  • Input Quantity — for editing statistics

  • Introduction Screen — onboarding screens and feature showcase screens

  • File Picker — for uploading injury report PDF
  • Syncfusion Flutter PDF Viewer — for viewing injury report PDF in-app
  • Dropdown Search — for searching injuries within injury dropdown
  • Google Sign In — Google SSO (implemented by Nathan)
  • Duration Picker — for recording player duration
Cross-Device Consistency

One of the challenges I encountered after starting the development of the application and doing some testing is maintaining cross-device consistency. While developing the application, I only considered testing it using one type of device, so when the tester tried it on different devices, some of the pages look different and are not responsive at all.

We want our users to use our application with ease no matter what device they decide to use, as we realised that it is important for users to have flexibility on what device they want to use depending on the task they want to do and the environment they are at.

Rooting from this problem, I decided to do my research on how to maintain cross-device consistency on Flutter applications, which can be read here:

I managed to develop PlayMetrix to be a cross-platform application with consistent look and feel throughout various devices, which is tested on desktop, tablet, and phone. Going forward, I will look more into the way the app looks on desktops, as currently, the menu is located at the bottom. If I were to further develop this, I will move the desktop navigation bar into a side bar instead of having it at the bottom of the page.

Folder Structure

Maintaining an organised workspace is really something that I overlooked when I started developing the app. Initially, I just had all my files in one folder, and as more screens are being developed, it really became hard to keep track of the files as they are all unorganised. I also had all the API calls and data models inside the screens, which became confusing when I started doing the calls across multiple screens.

After noticing this problem, I finally did a big refactor on the folder structure, organising it into several different folders based on its types and functionalities. This is definitely a really nice learning experience and something that I will keep in mind for my future projects.

Reflection

Overall, I am really satisfied with what I have developed for PlayMetrix. I managed to develop 48 functional screens for our app that can work across different devices. I am also happy that I got to learn new technologies and gained new skills in mobile development using Flutter.
