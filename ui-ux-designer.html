UI/UX Designer

Overview

During the concept release and the reflection sprints that follow after, I had the big responsibility of making and enhancing the prototype of the application. I had to visualise on how the features and the flow of the application should look like, while also maximising usability and the aesthetics of the application.

The steps I took were to:

  1. Develop the non-functional structure of each pages.
  2. Specify what is shown on each pages based on the user role.
  3. Develop the functionalities of each pages (connecting to backend).

Technical Challenges

  • Translating features of the application into pages in the application and enhancing its usability and visual appeal.
  • Designing the flow of the application to maximise usability for our users.
Planning PlayMetrix's UI

Before starting any UI design, I had to first get a deep probe into our users. As our application is tailored towards GAA teams, we had to do some research about the roles in the team that we were going to implement in our app, which are the:

  • Manager
  • Player
  • Physio
  • Coach

From this, we discovered that:

  • The manager is responsible for managing the whole team, such as organising trainings and choosing players for a match.
  • The coach helps the manager in doing that.
  • The physio keeps tracks of player injuries and helps the manager in deciding whether a player would be fit for playing.
  • The player will be able to inform a manager if they're attending a schedule, view their statistics.

These would be the main functionalities of our app that we will focus on, and now my responsibility is to transform these features into a UI that would be easy to use.

The steps I took to finally finish the prototype was to:

  1. Determine the pages needed for each user access level
  2. Determine the order of the pages and how it should flow
  3. Develop a wireframe of the application
  4. Develop a prototype on Figma based on the wireframe
  5. Enhance the prototype based on user testing feedback (Prototype V2)
  6. Further enhance the prototype based on external collaborator meetings and user testing feedback (Prototype V3)

After the initial release, we received more feedback and enhancements ideas, so I also made prototypes on this before actually developing it in the frontend. (UI Enhancements Prototype)

Wireframe

Creating an application's wireframe is, in my opinion, the hardest part of building a UI as I will need to build the screens and the flow from scratch. Firstly, I would need to decide on the forms I would need for each user, what data would be displayed for each screen, and how it will all tie together in a flow that makes sense for a user's workflow. After that, I would decide on the most "important" screens for the users and select them as the items for the bottom navigation bar.

Manager View

For the manager, one of their main priorities would be looking at the players at a glance and see who's available or not, so we wanted to reflect this in the UI using our traffic light system, which is represented by colours of the traffic light (red, yellow, green). For each player, information on their name, photo, and team number would be shown to make them easily identifiable by the manager, and their availability would be shown clearly in the UI through the border colour.

Player View

An important feature that a player would want is to see their statistics to see their playing time, matches played, etc. so we decided to include this page in the player's bottom navigation bar for them to get easy access to it whenever they need to access it. Scheduling is also an important aspect for our app whether it is for the player or the manager, so we also decided to put it on the bottom navigation bar for both user roles.

Physio View

A physio would also like to view players and edit their availabilities, as well as recording the player's injuries when they're injured. Just as in the manager view, the players list with each of the player's name, photo, and team number would be available for the physio to see for them to easily identify which player they would need to choose for editing the availability and/or recording the injury.

Prototype V2

After the initial feedback from the panel and several user testings, I had to make enhancements on the prototype during our first reflection sprint.

Some notable features that were added are:

  • 2 new users, coach and chairman
  • Managers can assign coaches to training and manage coaches
  • Coaches can view players and schedules
  • Chairman is responsible for setting up club and teams

Prototype V3

We had more feedbacks from external collaborators, user testing, and the panel, so more enhancements for the prototype are needed before finalising it.

Some notable features that were added/modified for the final version of the prototype are:

  • Remove the chairman access and club table, as it serves no purpose as the teams within the club will not correlate to each other in any way. The manager is now responsible of setting up their own teams.
  • Added an announcement feature for manager and coach to add updates on a certain schedule.
  • Added a way to record statistics of duration played for players.

UI Enhancements Prototype

After our initial release, we got several feedbacks from our user testing, which included:

  • Some users, especially those in the older age group have difficulties in getting onboarded in the application.
  • Notifications is an essential part of our app, so it would need to be highlighted more.

So for our next release, we decided to focus on improving the user's onboarding experience by adding introduction screens, enhancing our sign up flow, and having another set of screens showcasing the features for each user. I also redesigned the home screen and bottom navigation bar. The home screen now includes the menu items as well as the latest notifications, and I also added notifications to the bottom navigation bar for easier access.
